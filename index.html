01

Exhibit 01 · Premise

It knows what the artifact means

Most tooling teaches an agent how to run a command. Hermes teaches it what the output means — each artifact is mapped to its interpretation, the attacker behavior it implies, and its MITRE ATT&CK technique. The result reads like an examiner's notes, not a wall of raw tool output.

An artifact encyclopedia sits behind the agent so a memory image or an MFT record comes back as a conclusion with a confidence tier, already cited and cross-checked — ready for a human to review, not decode.

03

Exhibit 03 · Integrity model

Built so the analysis holds up

◎

Session canary

Every tool is validated at startup. A degraded tool drops the case to triage-only instead of quietly producing bad output.

⊟

Version pinning

Docker images pinned by digest, SIFT tools frozen. Reproducible today and next year — the same input yields the same result.

⇄

Cross-validation

Critical artifacts are confirmed by two independent tools. A delta over 5% blocks the finding until it's reconciled.

◈

Chain of custody

Every action is logged. Findings stay DRAFT until an examiner approves; the agent has no path to self-approve.

⇄

Correlation pass NEW

Each finding is cross-referenced against independent sources. Verdicts land in a read-only advisory summary — the examiner reviews it before sign-off.

03·b

Exhibit 03·b · Correlation pass NEW in v4.2.1

Every finding, independently checked

After analysis, before the examiner signs off, the correlation pass extracts every entity — IPs, hashes, filenames, domains — from each DRAFT finding and checks whether a different tool independently saw the same artifact. It proposes one of four verdicts per finding and writes them to a read-only advisory summary. It proposes; the examiner decides.

HIGH ▲
Corroborated
An independent source — a different tool — references the same entity.
LOW ●
Single-source
The entity appears only in the finding's own source. Examiner review.
REVIEW ■
Contradicted
A mechanical conflict — one filename, two different SHA-256 hashes.
— ◇
Unverified
No checkable entity, or nothing to check against. The honest default.
correlation-summary.txt · advisory — nothing modified
  F-niel-001  [CORROBORATED]  epxlorer.exe masquerading as explorer.exe
     corroborated by: TL-004(file:epxlorer.exe), EVID-001(file:epxlorer.exe)
  F-niel-002  [CORROBORATED]  C2 via fraudulent Telegram Bot API
     corroborated by: TL-005(ip:104.21.1.247)
  F-niel-004  [UNVERIFIED]    Analyst triage note
     no checkable entity could be extracted

  4 finding(s): 3 CORROBORATED · 0 SINGLE-SOURCE · 0 CONTRADICTED · 1 UNVERIFIED
  Advisory only. All findings remain DRAFT. Examiner decides.
Read-only, advisory only. The pass writes correlation-proposals.json and correlation-summary.txt and nothing else — it never edits findings, the timeline, the evidence, or the report. Verdict badges appear inline in the findings table of both report templates (timeline + data-first).

06

Exhibit 06 · Confidence tiers

Every finding is graded, not just stated

HIGH
Canary-validated, cross-validated by two tools, and the evidence OS is known.
MEDIUM
Validated on a single tool against a known OS build.
LOW
Canary passed, but the evidence OS could not be confirmed.
TENTATIVE
Canary failed — usable for triage direction only, never as evidence.
DEGRADED
Do not use for evidentiary work. A required tool is unavailable or unverified.
